
## 1) Conceito de Firewall Authentication

- A autenticação de firewall usa **usuários** e **grupos de usuários** para controlar acesso.
- É mais confiável do que confiar só em **IP** ou **tipo de dispositivo**.
- Depois da autenticação, o FortiGate aplica:
    - **políticas de firewall**
    - **perfis de segurança**
    - **permissões específicas por usuário/grupo**

---

## 2) Métodos de autenticação

### 2.1 Autenticação local

- Usuário e senha ficam **armazenados no próprio FortiGate**.
- É o método mais simples.
- Serve bem para ambientes menores.
- Pode incluir:
    - e-mail
    - telefone/SMS
    - **2FA**
    - associação com grupo
- Normalmente é validada via **captive portal**.

### 2.2 Autenticação remota

- As credenciais ficam em servidor externo:
    - **POP3**
    - **RADIUS**
    - **LDAP**
    - **TACACS+**
- Pode ser usada de duas formas:
    - usuário criado localmente no FortiGate, mas validado no servidor remoto
    - servidor remoto adicionado diretamente a um grupo

### 2.3 Duplo fator de autenticação

- Complementa a autenticação existente.
- Exige:
    - **algo que você sabe**: senha/PIN
    - **algo que você tem**: token/certificado

## 3) LDAP

### 3.1 Visão geral

- LDAP é um protocolo de diretório usado por sistemas como:
    - **Active Directory**
    - **Samba**
- Porta padrão:
    - **389** para LDAP
    - **636** para **LDAPS** (seguro)
- O “binding” é a etapa em que o servidor LDAP autentica o usuário.

### 3.2 Estrutura LDAP

- A estrutura segue uma árvore hierárquica:
    - **DC**: domínio
    - **OU**: unidade organizacional
    - **CN**: grupo/objeto
    - **uid** / **sAMAccountName**: identificação do usuário

### 3.3 Boa prática de configuração

- No FortiGate, o identificador **CN** pode trazer o nome completo, não o login.
- É mais recomendado usar:
    - **uid**
    - **sAMAccountName**
- Para conexão segura:
    - configurar **LDAPS**
    - importar certificado no FortiGate
- O bind type recomendado é **Regular**, com conta de menor privilégio.

## 4) RADIUS

### 4.1 Visão geral

- RADIUS oferece **AAA**:
    - Authentication
    - Authorization
    - Accounting
- Fluxo:
    - FortiGate envia **ACCESS-REQUEST**
    - servidor responde com:
    - **ACCESS-ACCEPT**
    - **ACCESS-REJECT**
    - **ACCESS-CHALLENGE**

### 4.2 Método de autenticação

- Quando o perfil está em **Default**, o FortiGate tenta nesta ordem:
    - **PAP**
    - **MSCHAP2**
    - **CHAP**
- A opção **Include in every User Group** inclui o servidor RADIUS em todos os grupos, mas isso só faz sentido em cenários bem específicos.

## 5) 2FA e FortiToken

### 5.1 O que é

- 2FA aumenta muito a segurança porque exige dois fatores.
- O usuário precisa provar:
    - conhecimento da senha
    - posse do token ou certificado

### 5.2 OTP

- OTP = senha de uso único.
- Formas de entrega:
    - **FortiToken físico**
    - **FortiToken Mobile**
    - **E-mail**
    - **SMS**
    - **FortiToken Mobile Push**

### 5.3 Sincronismo de tempo

- OTP depende de:
    - **seed**
    - **tempo**
- Por isso, o **NTP** precisa estar correto e sincronizado.

### 5.4 Atribuição do token

- Pode ser vinculado a:
    - usuário local
    - grupo
- Pode ser ativado por:
    - e-mail
    - SMS
    - token físico
    - FortiToken Cloud

## 6) Autenticação ativa e passiva

### 6.1 Autenticação ativa

- O usuário precisa digitar manualmente:
    - usuário
    - senha
    - às vezes token
- Normalmente acontece via **captive portal**
- Exemplos:
    - **POP3**
    - **LDAP**
    - **RADIUS**
    - **LOCAL**
    - **TACACS+**

### 6.2 Autenticação passiva

- Não exige tela de login do FortiGate.
- O FortiGate descobre a identidade automaticamente.
- Métodos:
    - **FSSO**
    - **RSSO**
    - **NTLM**

## 7) Uso em políticas de firewall

### 7.1 Origem da policy

- Pode-se usar na policy:
    - contas locais
    - contas remotas
    - usuários PKI/certificado
    - usuários FSSO
- Se o usuário pertencer ao grupo da policy, o acesso é permitido após autenticação bem-sucedida.

### 7.2 Protocolos necessários para autenticação ativa

- Para disparar o captive portal, a policy precisa permitir ao menos um destes:
    - **HTTP**
    - **HTTPS**
    - **FTP**
    - **TELNET**

### 7.3 DNS

- DNS pode ser permitido antes da autenticação.
- Isso ajuda os protocolos de autenticação funcionarem corretamente.
- O serviço DNS precisa estar explicitamente permitido na policy.

### 7.4 Políticas mistas

- Quando há políticas passivas e ativas:
    - se o usuário já foi autenticado passivamente, o FortiGate não pede credenciais novamente

## 8) Timeout de autenticação

- O timeout define quanto tempo o usuário pode ficar inativo antes de autenticar de novo.
- Padrão: **5 minutos**
- Modos:
    - **idle-timeout**: expira por inatividade
    - **hard-timeout**: expira após o tempo, mesmo com atividade
    - **new-session**: expira se não surgir nova sessão no período

- Em **Dashboard > Assets & Identities > Firewall Users** é possível:
    - ver usuários autenticados
    - listar logons FSSO
    - filtrar por grupo, IP, tempo, volume
    - desautenticar usuários manualmente
