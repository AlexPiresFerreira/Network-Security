
## 1) Criação de usuário local

### Caminho no FortiGate

- **User & Authentication**
- **User Definition**
- **Create New**

### Passo a passo

1. Acesse o menu **User & Authentication**.
2. Entre em **User Definition**.
3. Clique em **Create New**.
4. Selecione a opção de **usuário local**.
5. Preencha os dados principais:
    - **Name**: nome do usuário
    - **Password**: senha de acesso
    - Confirmação da senha
6. Salve a criação do usuário.
7. Se necessário, adicione esse usuário a um **grupo** de autenticação.
8. Use esse usuário em:
    - políticas de firewall
    - VPN SSL
    - autenticação administrativa
    - testes de acesso
### 1.2 Via CLI

```bash
config user local
    edit "<username>"
        set type password
        set passwd "<senha>"
        set status enable
    next
end
```

### 1.3 Adicionar usuário a um grupo

```bash
config user group
    edit "<nome_do_grupo>"
        set member "<username>"
    next
end
```

### 1.4 Verificação

```bash
show user local
show user group
```

---
### Observações importantes

- O usuário local fica armazenado no próprio FortiGate.
- É útil quando não existe integração com AD/LDAP/RADIUS.
- Pode ser usado como conta de emergência ou para testes.

### Resumo rápido

- Criar em **User Definition**
- Definir **nome + senha**
- Salvar
- Associar a grupo, se necessário

## 2) Instalação / cadastro de FortiToken

### Caminho no FortiGate

- **User & Authentication**
- **FortiTokens**
- **Create New**

### Passo a passo

1. Acesse **User & Authentication**.
2. Entre em **FortiTokens**.
3. Clique em **Create New**.
4. Informe os dados do token, como:
    - **serial number**
    - status do token
5. Salve o registro do token.
6. Associe o token ao usuário local desejado.
7. Valide se o token foi ativado corretamente.
8. Se o ambiente usar MFA, teste o login com:
    - senha do usuário
    - código temporário do token

### 2.2 Via CLI

```bash
# Registrar token
config user fortitoken
    edit "<serial_number>"
        set status active
    next
end

# Associar token ao usuário local
config user local
    edit "<username>"
        set two-factor fortitoken
        set fortitoken "<serial_number>"
        set email-to "<email@dominio.com>"
    next
end
```

### 2.3 Verificação

```bash
show user fortitoken
diagnose user fortitoken status <serial_number>
```

---
### Boas práticas

- Verifique se o token está **não utilizado** antes de associar.
- Confirme se o usuário já existe antes de vincular.
- Em produção, teste primeiro com um usuário piloto.

### Resumo rápido

- Cadastrar o token
- Associar ao usuário
- Validar autenticação de dois fatores

## 3) Configuração de LDAP

### Porta padrão
- **389**

### 3.1 Estrutura LDAP

- O material mostra a hierarquia típica:
    - `dc`
    - `ou`
    - `cn`
    - `uid`

### 3.2 Teste de autenticação LDAP e RADIUS


```bash
diagnose test authserver ldap <server name> <username> <password>
diagnose test authserver radius <server name> <scheme> <username> <password>
```

### Uso
- Integração com diretório para autenticação centralizada

### Caminho no FortiGate

- **User & Authentication**
- **LDAP Servers**

### Passo a passo

1. Acesse **User & Authentication**.
2. Entre em **LDAP Servers**.
3. Clique em **Create New**.
4. Informe os dados do servidor LDAP:
    - **Name** do servidor
    - **Server IP / FQDN**
    - **Porta**
        - normalmente **389** para LDAP
        - **636** para LDAPS
5. Configure o usuário de bind, se necessário:
    - **Distinguished Name**
    - **Password**
6. Defina o campo de identificação do usuário, conforme o diretório:
    - geralmente `sAMAccountName` em **Active Directory**
    - `uid` em **OpenLDAP**
7. Ajuste a base de busca:
    - **Base DN**
    - OU onde os usuários ficam armazenados
8. Execute o **test connection** para validar o acesso.
9. Salve a configuração.
10. Crie ou ajuste grupos de autenticação para usar os usuários LDAP nas políticas.

### 3.2 Via CLI

```bash
config user ldap
    edit "<nome_do_servidor>"
        set server "<IP_ou_FQDN>"
        set port 389
        set cnid "sAMAccountName"
        set dn "DC=empresa,DC=com"
        set type regular
        set username "CN=svc_fortigate,OU=ServiceAccounts,DC=empresa,DC=com"
        set password "<senha>"
        set secure ldaps          # opcional para LDAPS
        set port 636              # opcional para LDAPS
    next
end
```

### 3.3 Testar autenticação LDAP via CLI

```bash
diagnose test authserver ldap <nome_do_servidor> <usuario> <senha>
```

### 3.4 Debug em tempo real

```bash
diagnose debug application fnbamd -1
diagnose debug enable
```

### 3.5 Criar grupo de usuários LDAP

```bash
config user group
    edit "<nome_do_grupo>"
        set member "<nome_do_servidor_ldap>"
        config match
            edit 1
                set server-name "<nome_do_servidor_ldap>"
                set group-name "CN=VPN_Users,OU=Groups,DC=empresa,DC=com"
            next
        end
    next
end
```

---
### Pontos importantes

- LDAP centraliza autenticação no diretório corporativo.
- Muito comum em integração com **Active Directory**.
- Se usar LDAPS, confira certificado e porta segura.

### Exemplo de uso

- Usuários do domínio autenticando na VPN SSL
- Grupos LDAP aplicados em políticas de acesso
- Autenticação centralizada sem criar usuário local para cada pessoa

### Resumo rápido

- Criar o servidor LDAP
- Informar IP, porta e base DN
- Definir o identificador do usuário
- Testar a conexão
- Usar em grupos e políticas

## 4) Configuração de RADIUS

### Objetivo

O **RADIUS** é usado para autenticação externa centralizada, muito comum em:

- VPN
- MFA
- autenticação administrativa
- integração com sistemas de terceiros

### Passo a passo

1. Acesse a área de autenticação no FortiGate.
2. Adicione um novo servidor **RADIUS**.
3. Preencha:
    - **Name**
    - **Server IP**
    - **Shared Secret**
    - **Auth Port**
        - normalmente **1812**
    - **Accounting Port**
        - normalmente **1813**
4. Salve a configuração.
5. Teste a autenticação com um usuário válido.
6. Crie um grupo de autenticação para utilizar esse servidor RADIUS.
7. Aplique o grupo nas regras ou no serviço desejado.

### 4.2 Via CLI

```bash
config user radius
    edit "<nome_do_servidor>"
        set server "<IP_ou_FQDN>"
        set secret "<shared_secret>"
        set auth-type auto
        set acct-interim-interval 600
    next
end
```

### 4.3 Testar autenticação RADIUS via CLI

```bash
diagnose test authserver radius <nome_do_servidor> <metodo> <usuario> <senha>
```

> **Método:** `pap`, `chap`, `mschapv2`

### 4.4 Debug em tempo real

```bash
diagnose debug application fnbamd -1
diagnose debug enable
```

### 4.5 Criar grupo de usuários RADIUS

```bash
config user group
    edit "<nome_do_grupo>"
        set member "<nome_do_servidor_radius>"
    next
end
```

---
### Boas práticas

- Confirme se o servidor RADIUS está liberado na rede.
- Verifique se a secret coincide exatamente dos dois lados.
- Teste antes de colocar em produção.

### Resumo rápido

- Cadastrar servidor RADIUS
- Informar IP, secret e portas
- Testar autenticação
- Usar em grupo/política/VPN

#### FSSO 

### Comandos úteis


```bash
diagnose debug authd fsso
diagnose debug authd fsso server-status
diagnose wad user list
```

### Ideia geral

- Verificar autenticação de usuários
- Conferir status do servidor FSSO
- Listar usuários autenticados pelo WAD

### 4.1 Modos citados

- **DC Agent Mode**
- **Collector Agent-based Polling Mode**
- **Agentless Polling Mode**

### 4.2 Fluxo DC Agent Mode

- Usuário autentica no **Windows DC**
- DC Agent captura o evento
- Collector Agent repassa ao FortiGate
- FortiGate reconhece o usuário pelo IP

#### Portas citadas

- **UDP 8002**
- **TCP 8000**

### 4.3 Collector Agent-based Polling

- O Collector consulta os DCs periodicamente
- Usa:
    - **SMB TCP 445** por padrão
    - fallback com **TCP 135**, **TCP 139** e **UDP 137**

### 4.4 Tipos de polling

- **NetAPI**
- **WinSecLog**
- **WMI**

### 4.5 Comparação dos modos

- **DC Agent**
    - mais complexo
    - maior confiança
    - captura todos os logins
- **Polling**
    - instalação mais simples
    - pode ter atraso ou perder eventos dependendo do método

### 4.6 Advanced Settings

- **Citrix / Terminal Server**
- **RADIUS Accounting**
- **Syslog Servers**

### 4.7 Logs FSSO no FortiGate

- Eventos de login/logout
- NTLM authentication
- Recomenda-se nível mínimo:
    - **Notification**
    - ou **Information**

## Autenticação passiva
A **autenticação passiva** identifica o usuário sem exigir, necessariamente, uma ação direta de login no momento do acesso.

### Ideia principal
O firewall associa o usuário ao IP por meio de informações coletadas de forma indireta.

### Fontes comuns de informação
- eventos de logon em diretório
- serviços de autenticação
- agentes ou coletores
- integração com infraestrutura de identidade

### Benefícios
- Menos impacto para o usuário
- Permite políticas baseadas em identidade
- Ajuda a correlacionar usuário, máquina e tráfego

### Uso típico
- Aplicar políticas diferentes por usuário ou grupo
- Monitorar acessos de forma centralizada
- Melhorar rastreabilidade

## Collector Agent-Based Polling Mode Process
Esse modo usa um **collector agent** para reunir e distribuir informações de autenticação.

### Como funciona
1. O agente coleta eventos de logon/autenticação
2. Essas informações são consolidadas
3. O FortiGate consulta esses dados para identificar usuários
4. As políticas são aplicadas com base nessa associação

### Vantagens
- Centralização da coleta
- Boa integração com ambientes corporativos
- Facilita o mapeamento usuário/IP

### Pontos de atenção
- Depende da presença e funcionamento do coletor
- Pode exigir integração com diretórios e servidores específicos

---

## Remover o agent FSSO
O **FSSO** é o mecanismo de **Fortinet Single Sign-On**.

### O que significa remover o agent
- Eliminar o componente coletor/agente do cenário
- Trocar a estratégia de autenticação/identificação de usuários
- Simplificar ou alterar a arquitetura existente

### Motivos comuns
- Mudança de desenho da solução
- Substituição por modo sem agente
- Manutenção ou descontinuidade do componente

### Atenção
- Antes de remover, é importante validar o impacto nas políticas baseadas em usuário
- Pode afetar regras, logs e associação usuário/IP

---

## Agentless Polling Mode Process
Nesse modelo, o FortiGate obtém as informações **sem instalar agente nos endpoints**.

### Como funciona
- A identificação ocorre por consulta direta a fontes de autenticação ou diretórios
- O sistema coleta dados de forma centralizada
- Não depende de software em cada máquina do usuário

### Vantagens
- Menor dependência de clientes
- Mais simples de manter em alguns ambientes
- Reduz necessidade de instalação em endpoints

### Limitações
- Pode depender fortemente da qualidade da integração com os serviços consultados
- A visibilidade pode variar conforme a fonte de autenticação
