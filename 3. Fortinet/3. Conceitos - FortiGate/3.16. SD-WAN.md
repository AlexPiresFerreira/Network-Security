## 1) O que é SD-WAN

- **SD-WAN** é uma abordagem definida por software para direcionar o tráfego WAN com base em:
    - regras definidas pelo administrador
    - serviços e protocolos
    - aplicações
    - seleção dinâmica de link
- No FortiGate, isso faz parte do conceito de **Secure SD-WAN**, porque o roteamento é integrado aos recursos de segurança do FortiOS.
- O objetivo é:
    - melhorar o uso dos links
    - aumentar o desempenho das aplicações
    - reduzir custos
    - manter segurança no tráfego de saída

---

## 2) Underlay e Overlay

### Underlay

- São os **links físicos** fornecidos pelo provedor.
- Exemplos:
    - cabo
    - DSL
    - fibra
    - MPLS
    - 3G/4G/5G/LTE
    - ATM
- Características:
    - roteamento mais restrito
    - não adiciona segurança por si só

### Overlay

- São **links virtuais** construídos sobre o underlay.
- Exemplos:
    - **IPsec**
    - **GRE**
    - **IP-in-IP**
- Características:
    - roteamento mais flexível
    - segurança adicional

---

- **Underlay**

    - VLAN

    - FortiExtender
- **Overlay**
    - IPsec, incluindo **ADVPN**
    - GRE
    - IP-in-IP

## 4) Principais casos de uso

### 4.1 DIA — Direct Internet Access

- É o **acesso local à internet**.
- O tráfego sai diretamente pela filial, sem passar pelo hub/datacenter.
- Normalmente é usado para:
    - SaaS
    - navegação web
    - tráfego crítico que precisa de menor latência
- Benefícios:
    - menor atraso
    - melhor uso da banda WAN
    - menor dependência do hub

### 4.2 SIA / RIA — Secure Internet Access / Remote Internet Access

- É o cenário em que a internet sai **pelo HUB**.
- O tráfego da filial é trazido pelo overlay até o hub e então faz breakout para a internet.
- Vantagens:
    - centraliza inspeção de segurança
    - permite aplicar políticas de forma unificada
    - útil quando a empresa quer controlar melhor a saída

### 4.3 Cloud On-Ramp

- Direciona o tráfego para serviços em nuvem pelo melhor caminho.
- O objetivo é reduzir latência e melhorar acesso a aplicações SaaS/cloud.
- Pode usar:
    - conexão direta da filial para o cloud POP
    - túnel via hub
    - VPN em nuvem

### 4.4 ADVPN

- **ADVPN** permite comunicação direta entre spokes.
- O hub ajuda a negociar túneis de atalho.
- Vantagens:
    - menos latência entre filiais
    - menos carga no hub
    - melhor escalabilidade em topologias hub-and-spoke

## 5) Arquitetura e componentes do SD-WAN

A arquitetura destaca estes elementos:

- **Members**
    - interfaces que serão usadas para conduzir o tráfego
- **Zones**
    - agrupam membros em blocos lógicos
- **Performance SLA**
    - mede qualidade do link
- **SD-WAN Rules**
    - definem como o tráfego será direcionado
- **Routing**
    - fornece rotas válidas para o steering funcionar
- **Firewall Policies**
    - permitem o tráfego que foi selecionado pelas regras de SD-WAN

---

## 6) Members e Zones

- Os **members** podem ser interfaces físicas ou lógicas.
- As **zones** agrupam membros para facilitar a administração.
- A zona padrão é **virtual-wan-link**.
- A ideia é:
    - simplificar a configuração
    - permitir segmentação
    - reduzir a dependência de interfaces individuais em regras e policies

---

## 7) Performance SLA

- O **Performance SLA** monitora a qualidade dos links.
- Métricas principais:
    - **packet loss**
    - **latency**
    - **jitter**
- Ele pode ser medido de duas formas:
    - **ativamente**: probes enviados para servidores configurados
    - **passivamente**: com base no tráfego do membro
- Serve para decidir:
    - se o link está vivo
    - qual link é melhor para o tráfego

---

## 8) Regras de SD-WAN

- As regras são avaliadas **de cima para baixo**.
- A primeira regra que casar é aplicada.
- Uma regra SD-WAN define:
    - origem
    - destino
    - aplicação
    - serviço
    - internet service
    - membro ou zona preferida
    - SLA exigida
- Existe sempre uma **regra implícita**:
    - é usada quando nenhuma regra definida pelo usuário casar
    - segue o roteamento normal

---

## 9) Detecção de aplicações

- O FortiGate pode identificar o tráfego por:
    - aplicação
    - categoria de aplicação
    - internet service
    - grupo de aplicações
- Isso permite steering mais inteligente, como:
    - enviar Microsoft 365 por um link
    - enviar Salesforce por outro
    - tratar YouTube, Facebook e outras apps de forma separada

---

## 10) Roteamento no SD-WAN

- O SD-WAN precisa de **rota válida** para conseguir encaminhar tráfego.
- O comportamento pode usar:
    - rotas estáticas
    - rotas dinâmicas
    - ECMP
- Em muitos cenários:
    - uma rota default estática já é suficiente para DIA
- O roteamento pode referenciar:
    - um **member**
    - uma **zone**

---

## 11) Políticas de firewall

- Mesmo que a regra SD-WAN selecione o caminho, o tráfego ainda precisa de uma **firewall policy** para ser permitido.
- Na policy:
    - normalmente a interface de saída é uma **zone**
    - não se referencia membro individual diretamente em muitos cenários
- Também podem ser aplicados:
    - NAT
    - perfis de segurança
    - inspeção de tráfego

## 1) Performance SLA

- **Performance SLA** é usado para medir a saúde do link e decidir qual membro do SD-WAN deve ser usado.
- Os critérios principais são:
    - **packet loss**
    - **latency**
    - **jitter**
    - **link status**: up/down

### 1.1 Vários SLA targets

- É possível configurar **mais de um SLA target** em uma rule.
- Regra importante:
    - os targets precisam apontar para **servidores diferentes**
    - não funciona repetir o **mesmo servidor** no mesmo rule com múltiplos targets

### 1.2 Link status

- O status do link valida se o membro está **UP** ou **DOWN**.
- Exemplo do material:
    - **500 ms = 0,5 s**
    - **Failures = 5** → 
    - **Restore = 5** → 

### 1.3 Actions when Inactive

- A ação **Update static route** atualiza a tabela de rotas estáticas quando o link muda de estado.
- A observação do material indica que isso pode ser **desativado** se o link estiver down, dependendo do comportamento desejado.

## 2) SD-WAN Rule

### 2.1 Critérios de seleção / distribuição

- **Source IP**
    - mesma origem tende a sair sempre pelo mesmo link
- **Sessions**
    - balanceamento por sessões
- **Spillover**
    - usa o link até atingir os limites de entrada/saída definidos
- **Source-Destination IP**
    - pares de origem/destino iguais seguem o mesmo caminho
- **Volume**
    - divide o tráfego pelo número de pacotes/volume que passa por cada link

### 2.2 Estratégias de rule

- **Manual**
    - respeita a ordem configurada
    - só troca de link se o link cair
- **Best Quality**
    - escolhe o link de melhor qualidade
    - usa o parâmetro **`link-cost-threshold`**
    - valor padrão citado: **10%**
- **Lowest Cost (SLA)**
    - seleciona o link de menor custo conforme SLA
    - usa **`tie-break`** quando necessário:
    - `zone` (default)
    - `cfg-order`
    - `fib-best-match`
    - `input-device`
- **Maximize Bandwidth (SLA)**
    - usa todos os links que atendem ao SLA
    - o balanceamento pode ser ajustado com:
    - `set hash-mode <valor>`

### 2.3 Bandwidth da interface

- **Estimativa de banda** para o SD-WAN balancear carga:
    - `set estimated-upstream-bandwidth <valor>`
    - `set estimated-downstream-bandwidth <valor>`
- **Limite real da interface física / QoS**:
    - `set inbandwidth <valor>`
    - `set outbandwidth <valor>`

### 7.2 Leitura do exemplo

    - base de 20 pacotes
- **`fec-redundant 2`**

- **`fec-send-timeout 8`**
    - tempo da janela para envio dos pacotes redundantes

- O exemplo mostra melhora de perda:
    - antes: **27%** ou **12%** de perda
    - depois: queda para algo próximo de **1%**
- Em resumo:
    - **menor perda**
    - melhor recuperação em links ruins
    - útil para tráfego sensível

## 8) ADVPN

- **ADVPN** = **Auto-Discovery VPN**
- Serve para criar **shortcuts spoke-to-spoke** dinamicamente.

### 8.1 Ideia principal

- O tráfego começa pelo **túnel pai** via hub.
- O hub ajuda a negociar o **shortcut**.
- Depois disso, os fluxos subsequentes passam pelo **atalho** direto entre spokes.

### 8.2 Componentes mostrados

- **auto-discovery sender**
- **auto-discovery receiver**
- **auto-discovery forwarder**

### 8.3 Comportamento

- Primeiro pacote:
    - sai pelo túnel principal
    - dispara a negociação do shortcut
- Depois:
    - os pacotes seguintes passam pelo atalho
- O material também mostra:
    - shortcuts bem-sucedidos
    - shortcuts falhos
    - uso em topologias com **hub** e múltiplos links
